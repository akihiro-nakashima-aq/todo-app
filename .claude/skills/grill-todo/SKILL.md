---
name: grill-todo
description: Use when planning any change to this todo app — new features, schema changes, refactoring — and you want to stress-test the plan before writing code.
---

このtodoアプリへの変更計画について、共通認識に達するまで徹底的に質問してください。設計ツリーの各枝を一つひとつたどり、決定事項間の依存関係を順番に解決してください。各質問に推奨回答を添えてください。

質問は一度に一つずつ行うこと。

コードベースを調べれば答えられる質問は、質問せずに調査すること。

すべての質問が終わり共通認識に達したら、決定内容を `docs/spec/spec-<機能名>.md` としてWrite toolで保存してください。以下の構成で書くこと：

```markdown
# 仕様書: <機能名>

## 概要
（何を・なぜ作るか）

## DBスキーマ変更
（追加・変更するテーブル/カラム。変更なしの場合は「なし」）

## Server Actions
（追加・変更するaction名と処理内容）

## UI変更
（変更するコンポーネントとその挙動）

## 未解決事項
（実装中に判断が必要な残課題）
```

仕様書を保存したら、ユーザーに「修正はありますか？」と確認してください。修正がある場合は仕様書を更新し、再度確認してください。このサイクルを修正がなくなるまで繰り返してください。

修正がなくなったら、`docs/plans/plan-<機能名>.md` をWrite toolで作成してください。このファイルは実装の進捗追跡用です。処理が中断されても途中から再開できるよう、以下の形式で書くこと：

```markdown
# 実装計画: <機能名>

仕様書: `docs/spec-<機能名>.md`

## タスク

- [ ] （タスク1）
- [ ] （タスク2）
- [ ] （タスク3）
...
```

タスクは仕様書の内容から導出し、DBスキーマ変更→Server Actions→UIの順に並べること。実装を進める際はこのファイルのチェックボックスを完了したものから順に `- [x]` に更新してください。
